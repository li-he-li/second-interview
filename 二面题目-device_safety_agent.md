# 二面小项目：制造业设备安全操作 Agent

这是 AI Agent 开发实习岗位的二面小项目。我们不要求做得很大，但要求能运行、结构清楚、异常处理真实。

## 完成时间

建议 24-48 小时内完成。

## 提交方式

请提交以下内容：

1. GitHub 仓库链接或 zip 包。
2. README：说明启动方式、功能边界、技术选型、模块结构，以及哪些地方使用了 AI coding。
3. `.env.example`：不要提交真实 API key。
4. 至少 5 条测试样例和运行结果截图或日志。
5. 如果使用了大模型 API，请支持 mock/dry-run 模式，保证我们没有 API key 也能跑通主流程。

代码跑不起来会严重扣分。不要只提交页面截图，也不要只提交 Prompt。

## 项目背景

我们有制造业客户，后续会遇到设备说明、机械臂操作、安全规则、故障排查等场景。这个题目希望你做一个最小可运行的 Agent：

- 能根据设备知识库回答问题。
- 能识别用户意图。
- 涉及设备动作时，必须经过安全等级判断。
- 工具调用失败、检索不到资料、输入危险指令时，不能编造结果，要有明确 fallback。

## 项目目标

实现一个「制造业设备安全操作 Agent」最小可运行版本。

你可以用 CLI，也可以用 FastAPI。前端不是重点。

## 必做功能

### 1. 知识库

内置一个小型知识库，可以用 `knowledge/*.md` 或 `knowledge.json` 存储。

知识库至少包含：

- 设备基础说明。
- 安全操作规则。
- 常见故障和排查方法。
- 禁止执行或需要人工确认的操作。

示例知识内容可以自己设计，但要贴近制造业设备或机械臂场景。

### 2. 输入

支持用户输入一个问题或操作指令，例如：

- 设备报错 E42，应该怎么排查？
- 机械臂现在可以执行抓取动作吗？
- 把机械臂移动到 x=100, y=50, z=20，并抓取零件。
- 以最大速度直接移动到 x=9999, y=9999, z=9999。

### 3. 输出结构化 JSON

输出必须是结构化 JSON，至少包含以下字段：

```json
{
  "answer": "给用户的回答",
  "intent": "qa | status_check | device_action | unsafe_action | unknown",
  "sources": ["知识库来源 ID 或片段"],
  "confidence": 0.0,
  "safety_level": "L0 | L1 | L2",
  "need_human_approval": false,
  "tool_calls": [
    {
      "tool": "工具名",
      "input": {},
      "output": {},
      "status": "success | failed | skipped"
    }
  ],
  "final_action": "最终执行或建议执行的动作",
  "error": null
}
```

安全等级建议：

- `L0`：只读问答或状态查询，不涉及设备动作。
- `L1`：低风险动作，可以 dry-run，但需要记录日志。
- `L2`：高风险动作或参数异常，必须 `need_human_approval=true`，不能直接执行。

### 4. Mock 工具调用

至少实现 3 个 mock 工具：

```text
search_knowledge(query)
get_device_status()
execute_device_command(command, dry_run=True)
```

要求：

- `search_knowledge` 返回知识库命中的片段或空结果。
- `get_device_status` 返回设备状态，例如 online/offline/error/maintenance。
- `execute_device_command` 必须默认 dry-run，不允许真的执行危险动作。

### 5. 异常和失败处理

必须处理以下情况：

1. 检索不到相关资料。
2. 工具调用超时。
3. 工具返回异常。
4. 模型输出非法 JSON。
5. 用户输入危险指令。
6. 参数越界，例如坐标、速度、力度不合理。
7. 输入为空或意图无法判断。

遇到不确定或危险情况时，不要编造答案，要设置：

```json
{
  "need_human_approval": true,
  "safety_level": "L2"
}
```

### 6. 日志和 trace

必须有日志，至少记录：

- `trace_id`
- 原始输入
- 检索结果
- 工具调用输入输出
- LLM 原始输出或 mock 输出
- 最终 JSON
- 错误信息

日志可以写到控制台，也可以写到文件。

## 合格标准

我们会重点看：

- 是否能跑起来。
- 是否有清晰的数据流：输入 -> 检索 -> 意图判断 -> 工具调用 -> 安全审批 -> 输出。
- 是否有结构化输出和 schema 校验。
- 是否有证据来源，而不是模型凭空回答。
- 是否能处理失败情况：非法 JSON、API 超时、检索不到、输入为空、工具调用失败。
- 是否有日志和 trace，方便排查问题。
- 是否诚实说明哪些代码是 AI 生成，哪些设计和修改是你自己完成的。

## 可选加分

- 支持 FastAPI 接口。
- 支持简单的 Pydantic / JSON Schema 校验。
- 支持 mock LLM 和真实 LLM 两种模式。
- 支持将每次运行结果保存成 `runs/<trace_id>.json`。
- 支持简单的安全审批策略配置，例如 `safety_rules.json`。
