## 规则优先级

当规则冲突时，按以下顺序执行：

1. 用户当前明确指令
2. 项目内更具体的规范文件
3. 本 `CLAUDE.md/AGENTS.md`
4. 通用工程习惯

若用户当前指令存在明显安全、数据破坏、合规或工程风险，AI 必须指出风险并请求确认；不得静默执行高风险操作。

## AI 协作规范

### 1. 编码前先思考

以事实为依据，不臆断，不掩饰困惑，明确权衡，保持思想的活跃。

实现前需做到：

- 明确列出你的假设。如有不确定，主动提问。
- 若存在多种解读方式，全部列出，不要暗自选定一种。
- 若有更简单的方案，主动说明。必要时提出反对意见。
- 低风险任务可基于明确假设继续推进；高风险、不可逆、涉及数据删除/迁移/安全策略的任务，必须先确认。
- 人类和 AI 共同决策。如果人类的指令有重大问题，要及时纠正。
- 有自己的想法可以提出，认为审核后可执行。

### 2. 代码编写简洁优先

用最少的代码解决问题，不做任何预判性设计。

- 不实现需求以外的任何功能。
- 单次使用的代码不做抽象封装。
- 不添加未被要求的“灵活性”或“可配置性”。
- 不为未验证的假设做过度防御；但外部输入、数据库、网络、文件系统、安全边界必须有必要的异常处理。
- 若本可写 50 行代码却写了 200 行，重新精简。
- 自问：资深工程师会认为这段代码过于复杂吗？如果是，简化它。

### 3. 精准修改

只修改必要部分，只清理自己造成的冗余。

修改现有代码时：

- 不“优化”相邻代码、注释或格式。
- 不重构没有问题的代码。
- 遵循现有代码风格，即便你有不同写法。
- 若发现无关的无效代码，仅做提示，不直接删除。

当你的修改产生冗余内容时：

- 删除因你的修改而不再使用的导入、变量、函数。
- 除非明确要求，否则不删除原本就存在的无效代码。

检验标准：每一行被修改的代码都应直接对应用户需求。

### 4. 目标导向执行

定义成功标准，循环验证直至达标。

将任务转化为可验证目标：

- “添加校验” → “编写针对非法输入的测试用例，并让测试通过”
- “修复 bug” → “编写可复现问题的测试用例，并让测试通过”
- “重构 X” → “确保重构前后测试均能正常通过”

多步骤任务需简要列出计划，可用 openspec：

```text
1. [步骤] → 验证项：[检查内容]
2. [步骤] → 验证项：[检查内容]
3. [步骤] → 验证项：[检查内容]
```

清晰的成功标准支持自主迭代优化。模糊标准如“让它正常运行”会需要反复确认。

## Testing Rules

修复 bug 前：必须先构造复现。

修复后：必须保留回归测试。

### 1. 测试目标

测试必须回答：

- 是否复现问题
- 是否真正修复
- 是否影响旧功能

### 2. 测试资产

正式测试必须放在项目源码可见的 `tests/` 目录中，并优先使用项目主流测试框架。Python 项目默认 `pytest`。正式测试应能被一条稳定命令执行，并具备进入 CI 的条件。

`.temp/` 只用于一次性验证、排障脚本、临时实验和本地调查记录；其中内容不能作为“测试已覆盖”的依据。若某个 `.temp/` 脚本验证了真实缺陷或关键功能，任务完成前必须沉淀为 `tests/` 下的正式测试，或在文档中明确说明为何不能沉淀。

凡用于证明功能正确性、缺陷修复或回归保护的测试，最终应进入 `tests/`；无法进入时必须在测试报告说明原因。

涉及数据库、外部服务或本机环境的测试，必须提供可配置的隔离测试环境。不得让正式测试默认强依赖开发者本机数据库状态；至少应说明测试数据库、初始化数据、清理策略和环境变量配置。能用事务回滚、临时库、容器或专用测试库隔离时，优先隔离。

测试结果反馈给用户，并同步记录到 `docs/03-tests/`。`.temp/` 和 `logs/` 为运行时生成目录，需加入 `.gitignore`。

目录规范：

```text
tests/          → 正式自动化测试，可被 pytest/CI 执行
.temp/         → 一次性验证、排障脚本、临时实验
experiments/   → 实验功能
benchmarks/    → 性能测试
replays/       → 问题复现
logs/          → 运行日志
```

必须加入 `.gitignore`：

```gitignore
.temp/
logs/
outputs/
cache/
```

## 依赖和环境规则

- 新增依赖前必须说明原因。
- 不得为了省事全局安装依赖。
- 优先使用项目已有 lockfile、虚拟环境、容器或官方推荐环境。
- Python 项目不要在 base 环境中安装依赖或运行测试，必须单独创建虚拟环境。
- 如果项目已有依赖管理方式，优先遵循现有方式。

## 开发轨迹可见

每次完成一个阶段目标，或 debug，或测试，都要留档到 `docs/` 目录的对应分类中，做到整个迭代流程可见，AI clean 后可快速恢复上下文。

所有关键动作必须留档：

- feature
- debug
- review
- design decision
- benchmark
- breaking change

目录结构：

```text
docs/
├── index.md                  # 总索引：当前状态 + 模块地图 + 标签索引 + 时间线
├── 01-changelog/             # 功能变更日志：feature/phase 实现记录
│   └── YYYY-MM-DD.md
├── 02-debug/                 # 调试记录：bug 定位、修复过程
│   └── YYYY-MM-DD.md
├── 03-tests/                 # 测试报告：用例数、通过率、回归结果
│   └── YYYY-MM-DD.md
├── 04-reviews/               # Gatekeeper 审查记录：P0/P1 修复结论
│   └── YYYY-MM-DD.md
├── 05-design/                # 设计决策：架构讨论、技术选型记录
│   └── YYYY-MM-DD.md
└── 06-ai-assets/             # AI 特有资产
    ├── prompt-sessions/      # 关键 session 对话快照
    │   └── YYYY-MM-DD_session_summary.md
    └── model-version.log     # 模型版本、参数、环境变更记录
```

每个日志文件头部需包含 YAML frontmatter，供 AI 快速扫描：

```markdown
---
tags: [voice, stt, cli]
date: 2026-05-07
summary: 一句话描述本次记录内容
status: done
---
```

日志内容按时间顺序追加，格式：

```markdown
## [HH:MM] 标题
- 用户要求：xxx
- 改动内容：xxx
- 改动时间：xxx
```

改动记录不必深入到代码，总结即可。

`docs/index.md` 更新规则：

- 阶段性变更完成时必须更新。
- 新增测试报告、审查报告、设计决策时必须更新。
- 小型局部修复可只记录对应日志，必要时再更新索引。

## 自动审查流程

每个功能或者大的阶段，如 openspec 的 `## 1. Project Skeleton and Configuration` 完成后，自动触发 enterprise-code-gatekeeper 审查，不经手动指示。

触发时主 agent 必须在 prompt 中明确告知：

- 审查的文件路径
- 变更内容摘要
- 本次实现的功能目标
- 已执行的验证命令和结果

Gatekeeper 上下文隔离，无法看到主 agent 的对话历史。

流程：

```text
编码完成 → 自测通过 → 自动触发 gatekeeper 审查
                                    ↓
                              审查通过 → commit
                              审查不通过 → 修复 → 重新审查
```

通过标准：

- P0：必须修复。未清零则 block，不可 commit。
- P1：应当修复。反馈给用户，由用户决定是否跳过。
- P2/建议：记录但不计入通过条件。

兜底规则：

若 gatekeeper 不可用，主 agent 必须执行等价人工审查清单，并在 `docs/04-reviews/` 中说明未使用自动审查的原因。

人工审查至少覆盖：

- 安全和权限边界
- 数据一致性和并发风险
- 错误处理和异常路径
- 测试覆盖和回归风险
- 是否存在无关重构或范围扩大
- 是否有硬编码密钥、路径或环境依赖

## 禁止项

- Python 项目不要在 base 环境中安装依赖或运行测试，必须单开环境。
- 不得臆造项目事实、命令、接口或依赖。
- 不得在未确认风险的情况下执行破坏性命令。
- 不得把 `.temp/` 临时脚本直接当作正式测试覆盖证明。
- 不得在用户未要求时重构无关代码。
